# 05. マッチングアルゴリズム

> Day2-B 実装時に詳細を記載。現時点は方針メモ。

## 段階1: ルールベース + 重み付きスコアリング (Day2-B で実装)

### 入力

- 案件情報: 発地、着地、品目、必要車両タイプ、希望日時

### 処理フロー

1. **硬制約フィルタ** (該当しない候補は除外)
   - 車両タイプ適合
   - 稼働可能 (他案件と被らない)
   - 免許/資格 (品目別)

2. **スコアリング** (残った候補に対して)
   - `score = w1 * 過去顧客実績数 + w2 * 過去エリア実績数 + w3 * 平均評価 + w4 * (1 / 距離)`
   - 重み w1〜w4 はUIスライダーで顧客と調整可能に

3. **ランキング表示**
   - 上位10件、各スコア要素を内訳表示
   - 「なぜこのドライバー/車両が推奨されたか」を可視化

## 段階2 以降の検討メモ (未実装)

- 過去評価の指数移動平均(EMA)
- ドライバー×顧客の協調フィルタリング的スコア
- 複数案件の同時最適化 (OR-Tools)
- ML的アプローチ (XGBoost等で勝率予測)
- AWS Bedrock との連携検討

## 説明性 (重要)

顧客の中核要求として「AI的マッチング」とあるが、ベテラン配車係の判断を**置き換える**ものではなく**支援する**位置付け。
そのため、推奨理由の可視化(スコア内訳・重み調整可能) を必須とする。
