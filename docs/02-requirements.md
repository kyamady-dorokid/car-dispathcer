# 02. 顧客要求

> 詳細要求は今後追加。現時点で把握しているコア要求のみ。

## コア要求: 配車マッチングの最適化

配車に必要な情報を入力すると、**回収先事業場 × ドライバー × 車両タイプ** の最適な組合せを推奨する。

### 要素

1. **必要条件によるルールベースフィルタリング**
   - 車両タイプ適合 (例: 案件が4t必要なら4t車両のみ候補)
   - 免許/資格 (例: 危険物積載は有資格者のみ)
   - 稼働可能時間
   - エリア対応可否

2. **過去配車実績(約7000件)からのドライバー得意領域抽出**
   - エリア別の配車回数
   - 顧客別の配車回数
   - 経験のある荷物タイプ

3. **顧客からのドライバー評価**
   - 顧客→ドライバーへの評価データを反映
   - 評価の高いドライバーを優先

4. **AI的最適化処理**
   - 上記を総合してランキング表示
   - スコア内訳を可視化(顧客が納得できる説明性)

## 配車単位の仕様 (2026-06-01 確定)

骨格(Day1)完成後のヒアリングで、配車の単位が明確化された。

### 確定した前提

| 項目 | 内容 |
|---|---|
| **配車の単位** | 1案件 = 1車両 + 1ドライバー(**1日1車両=1案件**、1:1で確定) |
| **案件の中身** | **1案件 = 複数の集荷拠点を回って最後に1配送先へ(ミルクラン集荷型)** |
| **拠点の訪問** | 原則 1拠点 = 1日1回 |
| **拠点の重複** | 同一拠点を複数車両が回らない = 同日に同一拠点が複数案件に現れない |

### 解釈・影響(重要)

- 当初検討した「車両間で拠点を配分する VRP」ではない。**割当(案件→ドライバー/車両の推奨)は1:1のままでよく、Day1までの設計が活きる。**
- 新たに必要なのは「**1案件の中の複数集荷拠点を、どの順で回るか(案件内の巡回順=ミニTSP)**」。
- 最終配送先は1か所。集荷拠点が複数(2〜5程度を想定)。
- この「案件 = 複数stopの集合」が、将来の Google Maps / ZENRIN トラックルーティング差し替え時の入出力単位になる。

### 反映方針

- 本仕様は **docs に設計反映のみ(実装は後回し)**(2026-06-01 ユーザー判断)。
- スキーマ・シードの複数拠点化は別途タイミングを見て実施。Day2-A の地図実装は現行の単一発着モデルで進行可。
- データモデルの具体的な変更案は [04-data-model.md](04-data-model.md) の「複数拠点モデル(将来案)」を参照。
